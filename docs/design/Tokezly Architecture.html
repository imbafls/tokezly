Architecture · Greenfield · AI-first

Architecture.

How Tokezly gets built — from scratch, as our own project. Not a fork: the thing Tokezly actually does is the AI rewrite — turning the messy sentence you said into the polished one you meant, in real time — so the inference engine is the core, co-designed with transcription for low latency. Lightweight by construction (a Rust core under a thin web UI), capable by design.

00

The call

Recommendation

Build it ourselves — the AI rewrite is the product, not a plugin.

Tokezly's core trick isn't transcription, it's the transform: you speak a messy, half-formed sentence and Tokezly pastes the clean, properly-phrased version of what you meant — fillers gone, repetitions collapsed, mid-sentence self-corrections applied, grammar and structure fixed — in real time, by default. That's the category Typeless and Wispr Flow opened: intelligent dictation that listens for intent, not a verbatim transcript with a button bolted on.

That's exactly why we don't fork Handy. Handy is architected verbatim-first — Audio → VAD → ASR → paste. Making an always-on AI rewrite a first-class, low-latency stage means owning the whole pipeline and the AI engine: streaming the LLM, overlapping its prefill with transcription, keeping it warm, and tuning it for these bounded rewrites. Stapling that onto a verbatim transcriber fights its design and blows its latency budget.

We still keep the shell tiny (Rust core + thin web UI, no Electron) and we still learn from prior art — Handy for proven local-STT patterns, Typeless for the rewrite UX. We borrow the ideas, write our own code, and build the pipeline around the AI engine.

The transform — three levels of “fix what I said”

Same words in, different polish out. Clean is the default — the always-on text-shuffle that turns speech into writing. Verbatim and Prompt sit on either side of it, and all three run through the same optimized engine.

02

The AI inference core

This is the part we build deliberately, and the reason to go greenfield. One runtime hosts both the ASR engine and the refinement LLM behind a single compute scheduler — so they share the GPU instead of fighting over it, stay warm, and stream into one another. Everything else in the app serves this pipeline.

What the core owns

  • One backend, shared. A single compute abstraction (CUDA · Metal · Vulkan · CPU) used by ASR and LLM; GPU access is scheduled, never contended.
  • Warm by default. A model-lifecycle manager keeps the active ASR + LLM resident under a memory budget — the first keystroke pays no cold-start.
  • Streaming & overlap. ASR emits partials; the instant the transcript stabilizes, LLM prefill begins — hiding refinement latency behind transcription.
  • Quantized & cache-aware. int8 ASR, int4 (QAT) LLM, KV-cache reuse for the standing system prompt, pinned buffers.

Why this is the win

  • Refinement feels instant because most of its cost overlaps work you're already doing.
  • Memory stays predictable — one budget governs both models, no surprise OOM when Prompt Mode fires.
  • Swapping a model (ASR or LLM) is a config change, not a re-architecture.
  • Fully local by default; cloud is a provider behind the same interface.

Feasibility: fully-local STT + AI

Can the AI step stay on-device like transcription does? Yes — and it's the default. Gemma 3 4B is the right size: small enough to ship, fast enough to feel instant, capable enough for bounded rewrites.

Verdict

Fully Local by default. Gemma 3 4B (QAT int4) is the refinement engine.

Prompt-Mode tasks are bounded rewrites — commit messages, grammar fixes, tone shifts, list/JSON formatting — not long-form reasoning. A 4B instruct model handles these comfortably, and quantization-aware training keeps int4 quality within a few points of full precision. Running it in-process means audio and text never leave the machine. Cloud stays available as an opt-in for maximum quality.

How it plugs into the core

  • In-process, no daemon: bind llama.cpp from Rust (llama-cpp-2 / mistral.rs / candle) so there's nothing external to install — an Ollama sidecar is a valid quick path while bootstrapping.
  • One model manager: Gemma GGUFs download & cache beside the ASR models.
  • Shared scheduler: the LLM and ASR queue on the same compute backend — no GPU thrash.
  • Pure-CPU option: gemma.cpp for machines with no usable GPU.

Trade-offs to design for

  • Disk & RAM: a ~2.6 GB model + headroom; offer the 1B on <8 GB machines, detect at first run.
  • Cold start: keep the model warm while Tokezly runs (toggle in Settings).
  • Latency vs. cloud: local 4B is a touch slower than cloud; the live ms makes the cost honest.
  • Quality ceiling: the 12B or cloud opt-in covers gnarly instructions.
Privacy framing: default to fully local (audio + text on-device). Make any cloud refinement an explicit, clearly-labelled opt-in — a model marked Cloud and a one-time notice the first time data would leave the machine. Turn cloud on with an API key or a single sign-on (SSO) login that stores an auth token.

Extensibility — two traits

We design clean interfaces from day one. Adding a new ASR model or refinement provider means implementing one trait and registering it — the settings UI is generated from the registry.

ASR · transcription engine

pub trait AsrEngine { fn load(&mut self, model: ModelHandle) -> Result<()>; fn stream( &self, pcm: &[f32], on_partial: impl FnMut(Partial), ) -> Result<Transcript>; fn caps(&self) -> EngineCaps; // langs, gpu… }

REFINEMENT · Prompt-Mode engine

pub trait RefinementProvider { async fn rewrite( &self, text: &str, instruction: &str, ctx: PromptCtx, // selection, clipboard on_token: impl FnMut(Token), ) -> Result<String>; fn locality(&self) -> Locality; // Local | Cloud } // impls: GemmaLocal, OllamaLocal, CloudProvider
07

Bootstrapping & risks

Project shape

  • Rust workspace, crate-per-concern: audio · vad · asr · llm · core · app
  • Thin web UI (our Fluent-sleek frontend) over the command/event bridge
  • Per-OS bundles; models downloaded on first run, not bundled
  • Code-sign + notarize; in-app auto-update

Open questions / risks

  • Bundle vs. download the default Gemma GGUF (installer size vs. first-run UX)
  • GPU detection & graceful CPU fallback across vendors
  • Keeping models warm without idle VRAM waste
  • Overlay focus-steal on some Linux compositors — the capsule must never grab focus before paste
  • Cloud opt-in consent UX + key storage in the OS vault
